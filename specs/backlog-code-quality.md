# Code-quality backlog

Cross-cutting type-safety and lint-enforcement gaps surfaced by the 2026-06-06 quality audit. Scope is narrow: the project's hard rule is strong end-to-end typing across TypeScript, JavaScript, Python, and GDScript, with no hardcoded escape hatches (`any`, blanket `# type: ignore`, unscoped `eslint-disable`, `@ts-nocheck`) and no linters bypassed in CI or pre-commit.

These entries track places where a strict gate is configured but not enforced, or where a tree is exempted from type checking entirely. Feature-shaped work lives in [`backlog.md`](backlog.md); this file is exclusively code-health and toolchain enforcement. Each entry promotes into a numbered spec under `specs/` when work begins.

The 2026-06-11 reconciliation of spec 035 closed two of the original holes (the ESLint CI/pre-commit gate and the `packages/{models,codegen}` mypy gate both shipped in #112) and moved a third behind a trigger (the bpy-bound `ignore_errors` sweep now lives in [`GATED.md`](GATED.md)). The one entry below is what remains.

## Audit baseline (what is already clean)

So the next reader does not re-audit from scratch:

- **No blanket suppressions.** Every `# type: ignore` carries a specific error code (`[import-not-found]`, `[valid-type]`, `[arg-type]`); a grep for bare `# type: ignore` returns nothing. Every `eslint-disable-next-line` is single-line with a justification. The only `@ts-nocheck` is [`apps/photoshop/src/entry.ts`](../apps/photoshop/src/entry.ts) (vendored Adobe UXP starter shim, excluded from lint, documented). The only GDScript `gdlint: ignore` is one scoped line in a test.
- **tsconfig is maximally strict.** [`apps/photoshop/tsconfig.json`](../apps/photoshop/tsconfig.json) sets `strict`, `noImplicitAny`, `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`, `noPropertyAccessFromIndexSignature`, plus the unused-locals/params and `useUnknownInCatchVariables` checks.
- **mypy is strict-strict where it runs.** [`apps/blender/pyproject.toml`](../apps/blender/pyproject.toml), [`packages/validator/pyproject.toml`](../packages/validator/pyproject.toml), and now [`packages/models/pyproject.toml`](../packages/models/pyproject.toml) + [`packages/codegen/pyproject.toml`](../packages/codegen/pyproject.toml) carry `strict = true` (the models/codegen pair drops `disallow_any_explicit` under the pydantic plugin; the rest of the strict profile holds).
- **CI does not bypass gates.** No `continue-on-error`, `allow_failure`, or `|| true` anywhere in [`.github/workflows/`](../.github/workflows/). [`.pre-commit-config.yaml`](../.pre-commit-config.yaml) header states `--no-verify` is a bug. The `skipping` branches in `validate-schema` are "no fixtures present", not gate bypasses. ESLint now runs as a CI step and a pre-commit hook.

## The bundled proscenio_models wheel has no staleness gate

**What:** The Blender addon bundles a built copy of the source-of-truth model as [`apps/blender/wheels/proscenio_models-0.1.0-py3-none-any.whl`](../apps/blender/wheels/), and the writer imports `MeshElement` / `SpriteElement` / etc. from that wheel, not from `packages/models/src/`. The JSON Schema, Godot bindings, and TypeScript bindings each have a committed-match staleness test (`test_committed_schemas_match_emit`, `test_committed_artifacts_match_emit`, `test_committed_ts_bindings_match_emit`), but nothing checks that the bundled wheel reproduces from the current model. [`apps/blender/wheels/README.md`](../apps/blender/wheels/README.md) "When to bump" lists dependency and Python-ABI changes only, so a model field or shape change reads as not a wheel-rebuild trigger - even though the wheel is built locally from `packages/models/` and must be rebuilt on any shape change.

**Why it matters:** A contributor can edit `proscenio.py`, run `python -m proscenio_codegen all` (regenerating the schema, Godot, and TS artifacts - all gated), and ship a PR that looks complete while the bundled wheel still carries the old model. The Blender addon then exports the old wire shape. In PR #104 (multi-polygon export) the writer emitted the new `polygons` field, but the addon-side `MeshElement` loaded from the stale wheel rejected it with `extra_forbidden`; only a hand-written in-Blender test (`apps/blender/tests/operators/test_multi_polygon_export.py`) caught it, and that test had to self-skip when the wheel is stale rather than fail. CI installs into Blender's Python from `apps/blender/wheels/` (`ci.yml` "Install bundled wheels"), so a stale wheel silently degrades every Blender-side test against the old shape rather than going red. The wheel is the one downstream artifact of the source-of-truth model with no automated freshness check.

**Scope sketch:** Add a staleness check mirroring the schema/godot/ts ones - build `proscenio_models` to a temp dir, extract it, and assert the extracted `proscenio_models/*.py` (or the built model's `model_fields` / dumped schema) matches `packages/models/src/`. A byte-diff of the `.whl` is fragile (build timestamps and archive metadata vary run to run), so compare extracted source content or the in-memory schema, not the archive bytes. Wire it as a pytest under `tests/codegen/` and skip cleanly when the build backend is unavailable (mirroring the gdformat / npx skips elsewhere). Separately, add a `wheels/README.md` "When to bump" line: a shape change to any `packages/models/` model requires rebuilding and committing the `proscenio_models` wheel via `uv build packages/models --wheel --out-dir apps/blender/wheels`.

**Trigger to revisit:** the next `.proscenio` or PSD-manifest model field change, or when a Blender-side test passes against a stale wheel and a post-mortem asks why the shipped addon exported the old schema. Cheapest to land alongside the next schema-touching spec (schema-expressiveness), which rebuilds the wheel anyway.
