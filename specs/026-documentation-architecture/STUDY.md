# Documentation architecture: knowledge homes, docs IA, and the comment policy

Study to decide where each kind of project knowledge should live, redesign the Docusaurus information architecture by audience, define a routing policy for the ~2,900 existing code comments and docstrings, codify both in `.ai/`, and execute the result. This file maps the problem and tees up the decisions; it does not pre-decide them. Decisions land in [`decisions.md`](../decisions.md) and drive the `TODO.md` that follows.

## Problem

Three visible symptoms share one root cause: the project has no agreed map of which kind of knowledge belongs in which home.

1. The Docusaurus `guidesSidebar` conflates three audiences in one tree: the project pitch ("Proscenio Docs" landing), artist tutorials (Guides: basic / advanced), and developer reference (Project: architecture / features / comparison / deferred). A reader cannot tell who a page is for.
2. There is no canonical home for code-level design rationale. `decisions.md`, `specs/`, and the Docusaurus guides cover user-facing intent and the format contract; [`01-project/01-architecture.md`](../../docs/01-project/01-architecture.md) covers system altitude. Nothing owns line-level and module-level rationale, so it accretes as inline comments by default.
3. ~1,694 inline comments and ~1,201 docstrings do four different jobs with one tool. A prior audit flagged ~1,111 inline comments and ~249 docstrings as candidates for removal or relocation, but there is no rule for where each should go. That audit ran under a superseded all-or-nothing policy and has been removed; it is rebuilt under the routing taxonomy as the first execution step.
4. `.ai/conventions/` has no "when and what to comment" rule and no Docusaurus information-architecture rule, so whatever we clean up now drifts back.

Scope confirmed with the owner: one study covering all three axes; full audience-driven Docusaurus re-IA; execution included (docs migrated, comments routed and applied, conventions written); enforcement investigated, not just prose; all three languages (Python, GDScript, TypeScript), not Python alone. The decisions taken interactively are recorded in [Resolved decisions](#resolved-decisions-2026-06-10) below.

## Goals and non-goals

Goals:

- A single knowledge-home map: for every kind of knowledge (consumer how-to, format/API contract, system architecture, code-level rationale, local invariants, regression provenance, conventions) name its one canonical home.
- A Docusaurus IA redesigned by audience, with each existing page assigned a fate (keep / move / merge / cut).
- A comment-and-docstring routing policy: a decision rule a contributor or agent applies per comment.
- Both policies codified in `.ai/conventions/` (and `.ai/skills/` where agents need them), with enforcement investigated.
- Execution: the IA migration, the comment cleanup across all 131 affected files, and the convention writeup all land.

Non-goals:

- Rewriting documentation prose style. [`.ai/conventions/docs.md`](../../.ai/conventions/docs.md) already owns style (UI decorators, procedure steps, naming); this study owns architecture and the comment policy, and extends `docs.md` only with an IA section.
- Changing the schema, the `.proscenio`/manifest contract, or any runtime behavior.
- Editing autogenerated artifacts. `docs/content/*` (the schema reference the apiSidebar renders), the codegen outputs (Godot Resources, TypeScript types, the JSON Schemas under `packages/`), and the synced fixtures are generated from source - the comment policy and the IA changes never touch them. Relocating the Schema reference under Project is a `sidebars.ts` / navbar change, not a content move, and must leave the codegen pipeline intact. The `.gd`/`.ts` comment extraction excludes every generated path; only hand-authored source is routed.

The rebuilt inventory is the empirical test bed: the routing rule is only real if it routes the candidates into a defensible split.

Already covers the bpy-separation and dual-storage rationales.
- [`.ai/conventions/code.md`](../../.ai/conventions/code.md) - already states that `# Step N` comments signal a missing helper extraction (line 16), that bpy register-time constraints go in the module docstring (line 48), and that asserts document invariants (line 94). The comment policy extends this file; it does not contradict it.

## Axis A: Docusaurus information architecture

Redesign the site IA so each page has one audience and one altitude.

### A.1 Page audit (to produce)

Build a table of every page across the three sidebars, tagged on three dimensions:

- Audience: Consumer (artist using the tools), Maintainer (dev changing the code), Integrator (someone reading/writing the format).
- Altitude: Pitch, Tutorial, Reference, Architecture, Decision.
- Current value: Keep, Merge, Cut, Move.

The known pages: `00-guides` basic/advanced (Consumer, Tutorial), `01-project/{architecture,features,comparison,deferred}` (mixed), `02-blender-addon/*` (Consumer, Reference), `content/*` (Integrator, Reference, autogenerated). The audit fills the gaps and confirms the tags.

### A.2 Audience model (decision)

The core IA decision: how to slice the site by audience.

| Option | Shape | Trade-off |
| --- | --- | --- |
| Three audiences, three sidebars (recommended) | Consumer (landing + guides + addon reference), Maintainer (architecture + decisions + conventions mirror), Integrator (schema reference) | Clean separation matches the three real readerships; Maintainer sidebar becomes the home Axis B needs. Cost: one new sidebar, some pages move. |
| Two sidebars | User (consumer + integrator) vs Developer (maintainer) | Fewer sidebars; but integrator (schema) and consumer (artist) are genuinely different and would still feel mixed. |
| Keep three current sidebars, only clean Guides | Minimal | Lowest churn; but does not give code-level rationale a real home and the owner asked for a real redesign. |

Recommendation: three audiences. Decision left open.

### A.3 Page fates (decisions)

### A.4 Home for code-level cross-cutting rationale (decision, blocks Axis B)

The handful of cross-cutting, stable decisions that are too big for one line but too granular for nothing (XZ picture-plane convention; golden-compatible field ordering as a test constraint, not API). Options:

| Option | Where | Pin-ability |
| --- | --- | --- |
| Extend architecture.md sections (recommended for the ~2 genuine items today) | "Where to tread carefully" gains the XZ convention | Narrative; a comment can link the page but not a stable anchor |

| `.ai/conventions/code.md` only | Codebase-side, not the public site | Lives with the code; but not viewable on the docs site |

Decide whether such rationale belongs on the public site at all, or only in `.ai/`. This decision sets the destination that Axis B routes the few cross-cutting comments to.

## Axis B: comment and docstring routing policy

### B.1 Destination taxonomy

The closed set of homes a comment or docstring can route to:

- delete - restates code, padding, or refactor archaeology git already owns.
- keep-directive - tooling pragmas (`# type: ignore[code]`, `# noqa`); not prose, untouched.
- tighten-inline - a local invariant the next editor of this line must not break; kept but reduced to the constraint, not the narrative.
- to-assert - an invariant better expressed as an enforced `assert`/`raise` than prose (code.md already endorses this).
- to-test-link - bug-born "regression" knowledge; the why moves to a named regression test, the comment becomes a terse pointer to that test.
- to-module-docstring - module- or file-level purpose and constraints ("why this file exists, how it fits").
- to-architecture-or-conventions - cross-cutting, stable, re-litigated decisions; the high bar from Axis A.4.
- keep-apifact - an external-library fact or coordinate convention the code cannot show and no other home owns (terse).

### B.2 The routing rule (decision; the owner's explicit ask)

Define the decision tree a reader applies to one comment, in order:

1. Is it a tooling directive? keep-directive.
2. Does it only restate the adjacent code or narrate refactor history? delete.
3. Is it a local invariant that breaks if the line changes? Can it be an assert? to-assert, else tighten-inline.
4. Is the knowledge bug-born (a regression)? to-test-link (and if no guarding test exists, that is a missing test to write first).
5. Is it module/file purpose? to-module-docstring, when genuinely necessary (define the necessity bar: a one-line module whose name says it all does not need a docstring).
6. Is it cross-cutting, stable, and re-litigated? to-architecture-or-conventions (the rare case).
7. Is it an irreducible external-API fact? keep-apifact, terse.
8. Otherwise it is design narrative with a home elsewhere: move the substance, delete the comment.

Open sub-decisions: the exact necessity bar for module docstrings; whether to-assert is preferred over tighten-inline whenever feasible; the terse format for keep-apifact.

### B.3 The "belongs on the Docusaurus site" filter

A stricter gate than B.2 step 6: a piece of rationale reaches the public site only if it is (a) cross-cutting across modules, (b) stable, and (c) relevant to a consumer or to a maintainer's load-bearing-change decision. Everything else stays in code (docstring / test / assert) where it travels with the source and cannot rot independently. Reconcile this filter with the A.4 destination.

### B.4 Reference hygiene

- Regression comments cite a test by name, never a spec folder (docs.md link-rot rule). Define the citation format, for example `# regression: <symptom>; guarded by <test_name>`.
- Module docstrings state purpose and constraints, not change history.
- No comment references a `specs/NNN-*` path; if the rationale needs a written home, it is `decisions.md`, architecture.md, or a test.

### B.5 Empirical validation: re-tag the inventory (decision-gated)

With B.1 and B.2 fixed, rebuild the inventory across Python, GDScript, and TypeScript (generated paths excluded) and tag every comment and docstring with one destination from the taxonomy. Output: a per-file worklist with one destination each, plus the split counts. This is the proof the rule is operable and the input the execution waves consume. Surface any cluster the rule cannot cleanly route as a rule gap to fix before execution.

## Axis C: convention codification and enforcement

### C.1 Where each convention lands

- The routing rule (B.2), the destination taxonomy (B.1), reference hygiene (B.4): a new "Comments and rationale" section in [`.ai/conventions/code.md`](../../.ai/conventions/code.md), extending the existing function-level and assert guidance.
- The Docusaurus IA and the knowledge-home map (Axis A): a new information-architecture section in [`.ai/conventions/docs.md`](../../.ai/conventions/docs.md), distinct from the existing style content.
- Agent-facing summary: decide whether `.ai/skills/architecture.md` or a new `.ai/skills/` page is needed so agent sessions follow the same map.

### C.2 Enforcement investigation

`docs.md` already runs a drift-marker sweep and the pre-commit pipeline rejects common drift. Candidates to add, each scored on feasibility and false-positive cost:

- Block `# see specs/` / `# SPEC NNN` / spec-path references in code comments (high value, low false positive).
- Flag `# Step N` / `# Phase N` in code (code.md already calls these a smell; a hook makes it enforced).
- A heuristic prose-density or why-marker check (lower confidence, higher false positive; likely advisory only).

Decide which become pre-commit hooks, which stay advisory audits, and which are deferred.

### C.3 Agent alignment

Ensure the root `CLAUDE.md` / `.ai/README.md` point at the new sections so future sessions inherit the policy without re-deriving it.

## Sequencing and dependencies

- A.4 (rationale home) blocks B.3 (site filter) blocks B.5 (re-tag).
- B.1 and B.2 (taxonomy and rule) block C.1 (write conventions) and block all execution.
- Execution waves, lowest risk first:
  1. Docs IA migration (Axis A): move/merge/cut pages, rewrite `sidebars.ts`, fix inbound links.
  2. Re-tag inventory (B.5).
  3. Comment cleanup by category: first the unambiguous (delete echo/padding/archaeology, delete spec-reference comments and docstrings); then tighten-inline and to-assert; then to-module-docstring; then to-test-link (gated on the guarding test existing, which may spawn missing-test work); last the cross-cutting few to architecture/conventions.
  4. Write `.ai/conventions/` sections and land the enforcement hooks.
- The to-test-link subset is the long pole: a regression comment stays until its test exists, so this wave can surface a backlog of missing tests.

## Risks

- Scale: comment edits span 131 files. A single PR is too large to review and will conflict with other in-flight work. Stage in waves and small PRs, by category, not one sweep.
- Rot relocation: moving rationale to the site reproduces the exact failure mode if the bar is low. Mitigate with the strict B.3 filter and a strong default toward docstring/test/assert (homes that travel with code).
- Knowledge loss: deleting a regression comment before its guarding test exists loses the bug knowledge. Gate to-test-link on test existence.
- Link churn: the IA migration can break inbound doc links and the `repo-links` plugin; audit links as part of wave 1.
- Over-tightening: stripping a local invariant to its constraint can drop a load-bearing detail. tighten-inline keeps the constraint; when unsure, prefer to-assert so the runtime enforces it.

## Resolved decisions (2026-06-10)

The owner settled all six open questions interactively.

1. Site IA (A.2, A.5): three top-level navbar sections, not audience-labeled sidebars.
   - Guides: Basic, Advanced (consumer, learn by doing).
   - Project (low-level, for devs): Schema (Proscenio Character, PSD Manifest), Architecture, Comparison, Deferred.
   - Tools (high-level, intent per app): Blender Addon, Photoshop Plugin, Godot Plugin.
   The current `apiSidebar` (the autogenerated `content/` schema reference) is surfaced under Project > Schema; the current `blenderSidebar` becomes Tools > Blender Addon. This is a `sidebars.ts` and navbar restructure only - the generated `content/*` is never hand-edited and the codegen pipeline stays intact. Tools > Photoshop Plugin and Tools > Godot Plugin do not exist yet and are new hand-authored content this spec creates.
2. Features page (A.3): merge its unique content into the per-panel Tools > Blender Addon pages, then cut. One source per feature.
3. Comparison and Deferred (A.3): kept, under Project.
4. Code-level cross-cutting rationale home (A.4): extend architecture.md's "Where to tread carefully" section. No new conventions page; not `.ai/`-only.
5. Module-docstring necessity bar (B.2): a docstring only when the purpose or a constraint is not obvious from the module name plus its contents. Trivial modules get none.
6. Enforcement (C.2): high-confidence checks block at pre-commit (spec-path references in code, `# Step N` / `# Phase N`); the prose-density heuristic runs as an advisory report, never a commit gate.
7. Languages (scope): all three - Python, GDScript, TypeScript. The inventory grows `.gd` and `.ts` extractors (current tooling is Python-only via tokenize + ast) and the routing rule applies across all three. Every generated path is excluded from extraction.

Still-open micro-points, resolved during execution rather than now (non-blocking):

- The "Proscenio Docs" landing/pitch: site root intro, or a Guides entry. Decide in wave 1.
- Tools > Photoshop Plugin and Godot Plugin depth: full descriptive pages now, or stubs that grow, judged against the existing Blender Addon page depth. Decide in wave 1.
