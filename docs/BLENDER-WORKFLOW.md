# Blender workflow: rigging, animating, exporting, and surviving the iteration loop

How to author a Proscenio character in Blender, what survives saves and addon reloads, what does not, and the tradeoffs the addon makes when it sits between Photoshop and Godot.

This doc has two audiences. Sections marked **(Artist)** target the user authoring rigs and animations. Sections marked **(Contributor)** target someone touching the addon code. Most sections apply to both.

## The contract

Blender is the **hub** of the pipeline: it consumes Photoshop manifests on the inbound side and produces `.proscenio` files on the outbound side. The artist authors directly in Blender's native tools (weight paint, dopesheet, NLA, drivers) plus a thin Proscenio sidebar that exposes pipeline-specific knobs.

Two contracts run through Blender:

- Inbound: `psd_manifest.schema.json` v1 from the Photoshop UXP plugin (see [`docs/PHOTOSHOP-WORKFLOW.md`](PHOTOSHOP-WORKFLOW.md)).
- Outbound: `proscenio.schema.json` v1 to the Godot importer (see [`docs/GODOT-WORKFLOW.md`](GODOT-WORKFLOW.md)).

The user-facing rule: **work in Blender like Blender, use the Proscenio panel for pipeline knobs, export when ready. Re-export is non-destructive on the Godot side and idempotent on the Blender re-import side.**

## Layout (Artist)

```text
<your project>/
├── firebound.psd                source PSD
├── firebound/
│   ├── manifest.json            from UXP plugin
│   ├── images/                  per-layer PNGs
│   └── _spritesheets/           composed sprite_frame atlases
├── firebound.blend              hand-authored rig + animation
└── firebound.proscenio          generated by Blender writer (sticky path)
```

The `.blend` is yours. The `.proscenio` is regenerated; the `.blend` is the source of truth for everything authored Blender-side (rig, weights, animations, slot system).

## Lifecycle: what survives what

The Blender side equivalent of "what survives reimport" on the Godot side. Each row covers one lifecycle event and what state survives it.

| Event | Survives | Lost |
| --- | --- | --- |
| **Save `.blend` + reopen** | All Object data, PropertyGroups (PG), Custom Properties (CP), drivers, vertex groups, weights, actions, NLA tracks, sticky export path, slot Empties, atlas pack state, materials | Nothing - this is just normal Blender persistence |
| **Reload addon** (Get Extensions → Reload, or restart Blender) | All scene data, drivers, vertex groups, weights, actions, sticky path, slot Empties, atlas state. PG datablock is re-registered; existing PG values **survive** because the load_post handler re-hydrates them from CP | Operator running state (e.g. mid-Quick-Armature drag), modal handler internal state |
| **Re-import Photoshop manifest** | Per-mesh rotation, parenting, vertex weights, PG values, slot membership, animation tracks targeting the mesh by name | Mesh vertex data is replaced (UV + size from manifest); orphan meshes (no longer in manifest) are **left alone** but logged |
| **Uninstall addon** | Scene data, vertex groups, weights, actions, atlas materials, raw `proscenio_*` Custom Properties on objects. The data is preserved in CP form even with the addon gone | PG datablock disappears (Blender unregisters), so panel UI vanishes; drivers wired by Drive-from-Bone keep functioning (they are native Blender drivers) but you cannot re-edit them via the panel; help popups, validation operator, atlas packer all gone |
| **Blender version bump** | All standard Blender data | Addon may break on `bpy` API drift. Pin minimum version to 4.2 LTS; test on 4.5 LTS before bumping the minimum. CT2 history shows this is the most common failure mode for Blender addons (issues #92, #93, #95, #107, #109, #110, #111) |
| **Project folder move** | All `.blend` + sibling files. Sticky export path is stored as **absolute** today, so a folder move breaks the path; user must re-pick on next export | Sticky path | 

## PropertyGroup canonical, Custom Property fallback (Caveat)

[SPEC 005 D2](../specs/005-blender-authoring-panel/STUDY.md). The addon stores Proscenio metadata in two places:

- **PropertyGroup (PG)**: typed Python class registered on `bpy.types.Object.proscenio` and `bpy.types.Scene.proscenio`. Canonical. Edited via the Active Sprite / Active Slot / Export panels.
- **Custom Properties (CP)**: raw `proscenio_*` keys on the same Blender datablocks. Fallback for legacy reads, present for power users who want to script around the addon.

The two are kept in sync by a pair of handlers:

- `load_post`: hydrates PG from CP when the `.blend` opens. CP values authored externally arrive in the typed PG.
- `save_pre`: mirrors PG to CP before save. CP values stay current with PG edits made via the panel.

Implications for the artist:

| Action | Outcome |
| --- | --- |
| Edit via Active Sprite panel | PG updated immediately. CP synced on next save |
| Edit raw CP via Object Properties → Custom Properties mid-session | CP updated. **PG does not see the change until next `.blend` reopen** (load_post fires only on open). Panel reads still show old PG value |
| Save mid-session | save_pre fires; PG → CP mirror writes. CP edit you made earlier is **overwritten** by PG state |

**Default for artists**: edit through the panel. Reach for raw CP only if scripting around the addon, and reload the file to re-hydrate PG before resuming panel edits.

**Writer reads PG first, falls back to CP** at export time. So a CP-only edit (e.g. from a Python script) still propagates to `.proscenio` even if PG is stale.

## Authoring conventions (Artist)

The Proscenio sidebar tab opens with `N → Proscenio` in the 3D View. Inside, child panels expose every pipeline knob:

- **Active Sprite**: sprite type (`polygon` / `sprite_frame`), spritesheet metadata, texture region, Drive from Bone, Import Photoshop Manifest.
- **Active Slot**: slot anchor, default attachment picker, attachments list (when an `is_slot` Empty is selected).
- **Skeleton**: bone count, pose-mode helpers (Bake Pose, Toggle IK, Save Pose, Quick Armature, Create Slot).
- **Outliner**: sprite-centric flat list with substring filter and favorites.
- **Animation**: Action summary.
- **Atlas**: filename + Pack / Apply / Unpack.
- **Validation**: issue list with click-to-select.
- **Export**: sticky path, `pixels_per_unit`, Validate / Export / Re-export, Preview Camera.
- **Diagnostics**: smoke test.

Every subpanel header carries a status badge and a `?` button. The `?` opens an in-panel help popup with topic-specific content; same content lives in the dev skills.

### The naming-rule that bites

The writer matches **vertex group names to bone names exactly** ([SPEC 003 D7](../specs/003-skinning-weights/STUDY.md)). Renaming a bone in Edit Mode does **not** rename matching vertex groups; the writer silently drops weights for the renamed bone. Plan bone renames as a Blender-internal operation: rename bone, then rename matching vertex group.

## Recipes (Artist)

### 1. First rig from a Photoshop manifest

1. Run the UXP plugin in Photoshop, output to a folder.
2. In Blender, open or create the target `.blend`. Click **Import Photoshop Manifest** (Active Sprite subpanel), point at the manifest.
3. Planes appear at PSD positions, materials linked, single `root` bone created.
4. Add bones to the armature (Edit Mode, or **Quick Armature** for click-drag modal).
5. Parent meshes to the armature with `Ctrl+P → Armature Deform`.
6. Create vertex groups named after bones, paint weights in Weight Paint mode.
7. Set sprite type per mesh in Active Sprite.
8. Click **Validate**, then **Export**.

### 2. Iteration loop

Edit Blender → save → click **Re-export** in the Export subpanel. Sticky path is reused, no file dialog. Godot picks up the change on its next focus.

### 3. Rename a bone mid-project

The fragile recipe (analog of the cross-DCC rename caveat).

1. Note the old bone name.
2. In Edit Mode, rename the bone.
3. Rename matching vertex groups across every mesh that referenced the old bone (use Blender's outliner search to find candidates).
4. Update any drivers that referenced the old bone (Drivers Editor).
5. Update any slot whose `bone` parent referenced the old bone (Active Slot panel re-pick parent).
6. Rename matching tracks in any Action that targets the old bone.
7. Validate and re-export.

A bone rename is genuinely cross-cutting; there is no "rename refactor" tool today. If you author with renames in mind, name bones once and stick to it.

### 4. Add a sprite_frame variant after rigging

PSD-side: add the new frame inside the existing sprite_frame group, re-export. Blender-side: re-import the manifest. The mesh's sprite_frame metadata bumps to include the new index. Existing animation tracks continue to work; you can keyframe up to the new index.

### 5. Atlas pack / unpack flow

1. Materials with image textures are detected by the Atlas subpanel.
2. Click **Pack Atlas** to compose all per-sprite images into a single atlas image and update each sprite's `texture_region` to its slot in the pack.
3. Click **Unpack Atlas** later to reverse: each `texture_region` sub-rectangle becomes its own image again, materials updated.
4. **Apply Packed Atlas** is for the case where you packed externally; it just re-binds materials to the existing atlas file.

The packer is deterministic when input is deterministic (same images, same sizes). Used in fixtures for byte-equality goldens.

### 6. Multi-action animation

The writer iterates `bpy.data.actions`. Every action becomes one entry in the manifest's `animations[]` array. To author multiple animations:

1. In the Action Editor, create new actions with `+ New` or use the Action stash.
2. The Animation subpanel lists all detected actions.
3. The writer respects each action's frame range (start/end) to compute animation length.

NLA strips are **not** consumed by the writer today. To export an NLA-composed motion, bake it down to a single Action first.

## Features matrix

Status legend: `supported` shipped + tested in CI fixtures, `untested` plausibly works but no fixture covers it, `not supported` the writer skips it or errors.

| Blender feature | Status | Notes |
| --- | --- | --- |
| Single armature per scene | supported | the canonical case |
| Multiple armatures in scene | untested | writer assumes one; behavior on multi unclear, may pick first found or error. Avoid for now |
| Mesh objects parented to armature with `Armature Deform` | supported | drives skinning weights |
| Mesh objects parented directly to a bone (`Bone` parent type) | supported | rigid attachment, no skinning |
| Empty objects with `proscenio.is_slot=True` as slot anchors | supported | SPEC 004; `parent_type=BONE` and `parent_type=OBJECT` both honored |
| Vertex groups named after bones | supported | SPEC 003 D7 exact match |
| Vertex groups not matching any bone | supported (dropped with warning) | SPEC 003 D3 |
| Shape keys on sprite meshes | not supported | writer ignores; format v1 has no shape key concept |
| Mesh modifiers other than Armature | untested | writer reads evaluated mesh? Or base mesh? Behavior unclear; avoid heavy modifier stacks |
| Multiple materials on one mesh | untested | atlas detection may pick first material's image; multi-material mesh is fragile |
| Material with no image texture | untested | atlas detection skips? Errors? Avoid |
| Action with single F-Curve per channel | supported | the canonical case |
| Action assigned via Animation Editor (Action Editor) | supported | writer reads `obj.animation_data.action` and global `bpy.data.actions` |
| NLA strips composing motion | not supported | writer does not flatten NLA; bake to a single Action first |
| Linked / library override armature | untested | writer may not resolve linked datablocks; avoid for now |
| Drivers wired via Drive-from-Bone (5.1.d.1) | supported | drivers persist in `.blend`; survive save/reopen and addon uninstall |
| Drivers user-authored manually | untested | writer does not read drivers; they are runtime-only inside Blender |
| Custom drivers wired by other addons | untested | depends on whether the driven property is one the writer reads |
| Constraints other than IK (Copy Location, Track To, etc.) | untested | writer does not flatten constraints; pose at frame 0 may include constraint contribution depending on Blender's evaluation, avoid |
| IK constraints (`bpy.types.KinematicConstraint`) | untested at export | constraints are not exported (SPEC 000); pose includes their effect; bake before export if precise tracks needed |
| Drivers and constraints in pose evaluation | untested | writer reads pose at each keyframe via `bpy.context.evaluated_get`? - check before relying |
| Sprite mesh on Z != 0 plane | untested | writer assumes XZ plane (Y is depth in Blender world). A nonzero Y on a sprite likely encodes z-order via Z_EPSILON; large offsets will skew rendering |
| Color management (sRGB vs linear) on atlas image | untested | image is read as-is; non-sRGB images may render wrong in Godot |

When status is `untested`, **avoid the scenario in production** until a fixture or manual smoke test locks the behavior. Report friction so it can be promoted to a SPEC item.

## Validation lifecycle

Two checker layers ([SPEC 005 D6](../specs/005-blender-authoring-panel/STUDY.md)):

- **Inline** (cheap O(1) per redraw): status badges next to subpanel headers and inline error icons next to broken rows. Detects obvious issues without walking the scene.
- **Lazy** (heavy, on-demand): the **Validate** button walks the entire scene and produces an issue list with click-to-select. Errors block export; warnings inform.

The Export and Re-export operators **gate on Validate**. If any issue carries severity `error`, the operator aborts with a clear message. Warnings do not block.

When export blocks, fix the issues listed in the Validation subpanel and re-Validate. Common errors: no armature in scene, sprite missing required fields, vertex group name with no matching bone, atlas image not found.

## Tradeoffs (Artist + Contributor)

- **PG canonical, CP fallback (D2)**: typed editing wins; raw CP for legacy and scripting. Cost: mid-session raw CP edits do not propagate to PG until reload. Benefit: panel UI is type-safe and undoable.
- **Validate lazy + inline (D6)**: O(1) per redraw lets the panel feel snappy; heavy walk runs only on demand. Cost: a check that needs the full graph (e.g. cross-action consistency) waits for the user to click Validate.
- **Atlas in-tool (5.1.c.2) versus external packing**: the Blender packer is the supported path and runs on demand. External packers (TexturePacker, etc.) still work via Apply Packed Atlas, but lose the round-trip to source images that Unpack Atlas provides.
- **Sticky export path is per-scene + absolute**: simple to reason about; survives save/reopen but breaks on folder move. A relative-path mode is a candidate for a future polish wave.
- **Direct manipulation in the DCC pillar over proprietary modes**: every panel operation maps to a sequence of native Blender ops. No hidden state, no Proscenio-only mode. Cost: power features (Drive-from-Bone, Quick Armature) are convenience layers, not new paradigms.
- **Vertex group name = bone name (D7 SPEC 003)**: simplest possible matching rule. Cost: bone renames are cross-cutting (see recipe 3). Benefit: zero hidden mapping table; the writer behavior is inspectable.
- **Writer reads `bpy.data.actions` exhaustively**: every action goes to manifest. Pro: artist authors freely without manifest config. Con: stale actions left over in the file ship by accident; clean up before export.
- **Re-import idempotency via `proscenio.import_origin`**: per-mesh tag survives renames in Blender. See [`docs/PHOTOSHOP-WORKFLOW.md`](PHOTOSHOP-WORKFLOW.md) for the full caveat.

## Internals (Contributor)

The addon layout is documented in [`.ai/skills/blender-dev.md`](../.ai/skills/blender-dev.md). The lifecycle pieces relevant to this doc:

- **Registration**: `__init__.py` chains submodule `register()` / `unregister()`. Each operator / panel / property module ships its own `_classes` tuple. Reload depends on clean unregistration; leaks break reload.
- **Handlers**: `properties/_handlers.py` registers `load_post` (CP → PG hydrate) and `save_pre` (PG → CP mirror). Both are persistent (`@bpy.app.handlers.persistent`), so they survive `.blend` reload. Driver and modal handler cleanup happens in `unregister()`.
- **Driver registration**: Drive-from-Bone wires native Blender drivers, then Blender's own driver runtime owns them. Addon uninstall does **not** strip the drivers (they are scene data, not addon-owned).
- **Modal operators**: Quick Armature uses a modal viewport handler. Cancel via Escape; partial state (mid-drag) is discarded cleanly. The modal contract requires the operator to register and unregister cleanly across reloads.
- **`bpy.types.Object.proscenio` PropertyGroup**: registered in `properties/__init__.py`. Re-registration on reload re-uses existing CP via load_post hydration, so user data survives.
- **Validator subpackage** (`core/validation/`): each scope has its own module (`active_sprite`, `active_slot`, `export`, `_shared`). Validators are pure functions over `SimpleNamespace`-style mocks; pytest exercises them headless. `Validate` operator orchestrates the suite over the live scene.
- **Writer package** (`exporters/godot/writer/`): per [SPEC 009](../specs/009-code-modularity/STUDY.md). Public entry is `export(scene)`. Internal modules (`scene_discovery`, `skeleton`, `sprites`, `slots`, `slot_animations`, `animations`) are pure-Python and unit-tested.

For the full register / unregister cycle and how reload survival is guaranteed, walk `apps/blender/__init__.py` plus `properties/_handlers.py`.
