# Blender workflow

How to author a Proscenio character in Blender, what survives saves and addon reloads, and what the addon does between Photoshop on one side and Godot on the other.

## The contract

Blender is the **hub** of the pipeline. It consumes the PSD manifest on the inbound side and produces `.proscenio` on the outbound side. Authoring uses Blender's native tools (weight paint, dopesheet, NLA, drivers) plus a thin **Proscenio** sidebar tab that exposes the pipeline-specific knobs.

Rule of thumb: work in Blender like Blender, use the Proscenio panel for pipeline-specific knobs, export when ready. Re-export is idempotent.

## Project layout

```text
<your project>/
├── firebound.psd               source PSD
├── firebound/
│   ├── manifest.json           emitted by the Photoshop UXP plugin
│   ├── images/                 per-layer PNGs
│   └── _spritesheets/          composed sprite_frame atlases
├── firebound.blend             hand-authored rig + animation
└── firebound.proscenio         generated by the writer (sticky path)
```

The `.blend` is yours. The `.proscenio` is regenerated; the `.blend` stays authoritative for everything authored in Blender (rig, weights, animations, slot system).

## What survives what

| Event | Survives | Lost |
| --- | --- | --- |
| Save `.blend` + reopen | every Object datablock, PropertyGroups, Custom Properties, drivers, vertex groups, weights, actions, NLA tracks, sticky export path, slot Empties, atlas state, materials | nothing - normal Blender persistence |
| Reload addon | every scene datablock; PropertyGroup values re-hydrate from Custom Properties via the `load_post` handler | running operator state (e.g. mid-Quick-Armature drag), modal internal state |
| Re-import PSD manifest | per-mesh rotation, parenting, vertex weights, PropertyGroup values, slot membership, animation tracks targeting the mesh by name | mesh vertex data is replaced (UV + size from manifest); orphan meshes (no longer in manifest) are left alone but logged |
| Uninstall addon | scene data, vertex groups, weights, actions, atlas materials, raw `proscenio_*` Custom Properties | the typed PropertyGroup datablock disappears, so the panel UI vanishes; drivers wired by Drive-from-Bone keep working (they are native Blender drivers) |
| Blender version bump | every standard Blender datablock | addon may break on `bpy` API drift; test on the next LTS before bumping the floor |
| Project folder move | every `.blend` + sibling files | sticky export path is stored as **absolute** today and breaks on a folder move; user re-picks on next export |

## PropertyGroup canonical, Custom Property fallback

The addon stores Proscenio metadata in two places:

- **PropertyGroup** - typed Python class on `bpy.types.Object.proscenio` and `bpy.types.Scene.proscenio`. Canonical. Edited via the sidebar.
- **Custom Properties** - raw `proscenio_*` keys on the same data blocks. Fallback for legacy reads and power users who script around the addon.

Two persistent handlers keep them in sync: `load_post` hydrates PropertyGroup from Custom Properties on file open, `save_pre` mirrors PropertyGroup back to Custom Properties before save.

Practical implication: edit through the panel. Mid-session raw Custom Property edits do not propagate to the PropertyGroup until the next reopen, and a save will overwrite them with the stale PropertyGroup value. The writer reads PropertyGroup first and falls back to Custom Properties, so a script-only edit still reaches `.proscenio` even if the PropertyGroup is stale.

## Authoring panel

Open with **N -> Proscenio** in the 3D View. Subpanels:

- **Active Sprite** - sprite type (`polygon` / `sprite_frame`), spritesheet metadata, texture region, Drive from Bone, Import Photoshop Manifest.
- **Active Slot** - slot anchor, default attachment picker, attachments list (when an `is_slot` Empty is selected).
- **Skeleton** - bone count, pose-mode helpers (Bake Pose, Toggle IK, Save Pose, Quick Armature, Create Slot).
- **Outliner** - sprite-centric flat list with substring filter and favorites.
- **Animation** - Action summary.
- **Atlas** - filename + Pack / Apply / Unpack.
- **Validation** - issue list with click-to-select.
- **Export** - sticky path, `pixels_per_unit`, Validate / Export / Re-export, Preview Camera.
- **Diagnostics** - smoke test.

Each subpanel header carries a status badge and a `?` button. The `?` opens a topic help popup.

### The naming rule that bites

The writer matches **vertex group names to bone names exactly**. Renaming a bone in Edit Mode does not rename matching vertex groups; the writer silently drops weights for the renamed bone. Treat bone renames as a Blender-internal operation: rename the bone, then rename the matching vertex group across every affected mesh.

## Recipes

### 1. First rig from a Photoshop manifest

1. Photoshop: run the UXP plugin, export to a folder.
2. Blender: open the target `.blend`, click **Import Photoshop Manifest** (Active Sprite subpanel), point at the manifest.
3. Planes appear at PSD positions, materials linked, single `root` bone created.
4. Add bones to the armature (Edit Mode, or **Quick Armature** for a modal click-drag).
5. Parent meshes to the armature with `Ctrl+P -> Armature Deform`.
6. Create vertex groups named after bones, paint weights in Weight Paint.
7. Set sprite type per mesh in Active Sprite.
8. **Validate**, then **Export**.

### 2. Iteration loop

Edit Blender → save → **Re-export** in the Export subpanel. Sticky path is reused, no file dialog. Godot picks up the change on its next focus.

### 3. Rename a bone mid-project

Cross-cutting; there is no rename-refactor tool today.

1. Note the old bone name.
2. Edit Mode, rename the bone.
3. Rename matching vertex groups across every affected mesh.
4. Update any drivers that referenced the old bone.
5. Update any slot whose `bone` parent referenced the old bone.
6. Rename matching tracks in any Action that targets the old bone.
7. Validate and re-export.

Naming bones once and sticking to it is much easier than renaming later.

### 4. Add a sprite_frame variant after rigging

PSD-side: add the new frame inside the existing sprite_frame group, re-export. Blender-side: re-import the manifest. The mesh's spritesheet metadata bumps to include the new index; existing animation tracks continue to work and you can now keyframe up to the new index.

### 5. Atlas pack / unpack flow

1. Materials with image textures are detected by the Atlas subpanel.
2. **Pack Atlas** composes per-sprite images into a single atlas and updates each sprite's `texture_region`.
3. **Unpack Atlas** reverses the operation: each region becomes its own image again, materials updated.
4. **Apply Packed Atlas** is for the case where you packed externally; it rebinds materials to the existing atlas file.

The packer is deterministic when input is deterministic - used in CI fixtures for byte-equality goldens.

### 6. Multi-action animation

The writer iterates `bpy.data.actions`. Each action becomes one entry in the manifest's `animations[]`. Create actions in the Action Editor; the writer respects each action's frame range. NLA strips are **not** consumed - bake to a single Action first.

## Feature support

| Feature | Status |
| --- | --- |
| Single armature per scene | supported (canonical case) |
| Mesh objects parented to armature (Armature Deform) | supported (drives skinning weights) |
| Mesh objects parented directly to a bone (Bone parent type) | supported (rigid attachment) |
| Empty objects with `proscenio.is_slot=True` as slot anchors | supported (`parent_type=BONE` and `parent_type=OBJECT` both honoured) |
| Vertex groups named after bones | supported (exact-match rule) |
| Vertex groups not matching any bone | supported (dropped with console warning) |
| Action with single F-Curve per channel | supported (canonical case) |
| Drivers wired via Drive-from-Bone | supported (survive save/reopen and addon uninstall) |
| Shape keys on sprite meshes | not supported (writer ignores; format has no shape key concept) |
| NLA strips composing motion | not supported (bake to a single Action first) |

Scenarios outside this table (linked / library override armatures, multi-material meshes, color management quirks, constraints other than IK) are not exercised by CI fixtures. The safe path is to flatten the scenario before authoring rigs on top of it. If a real workflow surfaces friction, log it so it can be promoted to a SPEC.

## Validation lifecycle

Two checker layers:

- **Inline** - O(1) per redraw; status badges next to subpanel headers and inline error icons next to broken rows. Detects obvious issues without walking the scene.
- **Lazy** - heavy, on-demand; the **Validate** button walks the entire scene and produces an issue list with click-to-select.

Both **Export** and **Re-export** gate on the lazy validator. If any issue has severity `error`, the operator aborts. Warnings do not block.

Common errors: no armature in the scene, sprite missing required fields, vertex group with no matching bone, atlas image not found.
